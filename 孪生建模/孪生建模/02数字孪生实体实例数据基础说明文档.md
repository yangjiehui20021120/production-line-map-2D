# 动车组侧墙产线数字孪生数据基础说明文档

## 文档目标

本文档旨在为智能应用设计人员（包括Gemini、ChatGPT等大模型）提供动车组侧墙产线数字孪生系统的完整数据基础理解,确保后续开发的业务优化应用能够充分利用这套体系完整、具有上下文的建模数据。

**适用场景**:
- 产线计划可行性验证
- 产线价值流分析  
- 产线自制品管控
- 侧墙生产质量提升
- 其他基于数字孪生的业务优化应用

---

## 一、数据体系架构概览

### 1.1 核心设计理念

本数字孪生系统采用**NGSI-LD标准**构建,基于以下核心理念:

1. **语义化建模** - 所有实体采用统一的URN标识符和标准化的属性/关系结构
2. **三锚点数据模型** - 运行时数据通过 TwinObject(孪生体) × Scene(场景) × Modality(模态) 三维锚定
3. **分层建模** - 从静态资源到动态业务,从工艺定义到执行实例的完整层次
4. **全流程追溯** - 从MBOM工艺定义到实际执行场景,再到测量数据的完整追溯链

### 1.2 数据体系分层结构

```
┌─────────────────────────────────────────────────────────┐
│                    运行时数据层                           │
│  ModalData: 承载所有测量值、观测值、质量数据              │
│  (通过三锚点定位: Twin × Scene × Modality)                │
└─────────────────────────────────────────────────────────┘
                            ↑
┌─────────────────────────────────────────────────────────┐
│                    执行场景层                             │
│  Scene: 业务过程实例 (Takt/Process/Step级别)             │
│  记录实际发生的工序、时间、人员、资产                      │
└─────────────────────────────────────────────────────────┘
                            ↑
┌─────────────────────────────────────────────────────────┐
│                    工艺定义层                             │
│  MBOM: 制造工艺路线 (Route→Takt→Process→Step)            │
│  定义标准工艺流程、资源要求、测量规范                      │
└─────────────────────────────────────────────────────────┘
                            ↑
┌─────────────────────────────────────────────────────────┐
│                    资源配置层                             │
│  TwinObject: 物理/逻辑资源 (设备、人员、工件、产品)        │
│  Modality: 数据字典 (定义可测量的数据类型)                │
│  ModalityBinding: 将Modality绑定到具体TwinObject          │
│  OrgUnit, Role, Assignment: 组织与人员关系                │
└─────────────────────────────────────────────────────────┘
```

## 二、核心数据实体详解

### 2.1 TwinObject (数字孪生对象)

**定义**: 代表产线中的所有物理或逻辑实体

**核心属性**:
- **twinType**: 分为构成性(Constituent)和流转性(Transitional)
  - 构成性: 产线的固定资源(设备、工位、台位、人员)
  - 流转性: 随生产流动的对象(工件、产品、物料)

- **subType**: 细化分类
  - Robot, Welder, Positioner, Crane (自动化设备)
  - Scanner, QC_Tool, Fixture (质量与工装)
  - Person (人员)
  - Workpiece, Product, Material (流转对象)
  - Station, Position (产线布局)

- **functionCategory**: 生产功能分类(F1/F2体系)
  - F1: 主价值流功能
  - F2: 支持性功能

**特定子类型扩展**:
- **Station**: 工位实体,包含位置信息、所属组织、能力标签
- **Position**: 台位实体,工位内的具体作业点,包含偏移坐标、台位类型、物流连接
- **Person**: 人员实体,包含技能、资质、关联角色

**数据驱动能力**:
- 设备资产管理与状态监控
- 产线布局与工位配置
- 人员技能匹配与任务分配
- 工件流转追溯与在制品管理

---

### 2.2 MBOM (制造物料清单)

**定义**: 制造工艺路线的标准化定义

**层级结构**:
```
MBOM Root (产品级)
  ↓
Route (工艺路线)
  ↓
Takt (节拍段/大工段)
  ↓
Process (工序)
  ↓
Step (工步)
```

**核心内容**:
- **Route**: 定义产品的完整制造路径,包含所有Takt序列
- **Takt**: 节拍段,包含多个Process,定义标准节拍时间
- **Process**: 工序,包含多个Step,定义执行工位、资源类别
- **Step**: 工步,最细粒度,定义:
  - 标准工时(stdTime)
  - 资源类别要求(resourceClasses)
  - 执行工位/台位(stationCode, positionCode)
  - 测量要求(measurements): 定义需要采集的Modality及质量规则
  - 工艺文档引用(stepDocuments)

**数据驱动能力**:
- 工艺标准化与版本管理
- 产线计划排程(基于标准工时)
- 资源需求计算
- 质量检测点配置
- 工艺文件追溯

---

### 2.3 Scene (场景/执行实例)

**定义**: 业务过程的实际执行记录

**分层级别**:
- **Takt级Scene**: 记录一个节拍段的实际执行
- **Process级Scene**: 记录一个工序的实际执行  
- **Step级Scene**: 记录一个工步的实际执行(最细粒度)

**核心信息**:
- **refMBOM**: 引用对应的MBOM定义(建立实际与标准的关联)
- **timeFrame**: 实际执行时间(start, end, actualDuration)
- **location**: 执行位置(lineCode, zoneId, stationCode)
- **status**: 执行状态(Planned/InProgress/Completed/Failed)
- **inputWorkpiece / outputWorkpiece**: 输入输出工件(建立工件流转链)
- **inputMaterials**: 消耗的物料清单
- **involvesAsset**: 参与的资产(设备、人员)
- **stepLog**: 工步执行日志(人员、角色、时长、结果)

**数据驱动能力**:
- 实际生产过程记录
- 工件流转追溯(通过inputWorkpiece/outputWorkpiece构建链)
- 资源使用分析
- 工时对比分析(实际vs标准)
- 异常事件记录
- 人员履职追溯

---

### 2.4 Modality (模态/数据字典)

**定义**: 定义一种可观测数据的业务类型及采集规范

**核心维度**:
- **category**: 数据类别
  - process: 工艺过程参数(焊接电流、温度、速度)
  - quality: 质量检测数据(尺寸、轮廓、缺陷)
  - geometry: 几何测量(坐标、偏差)
  - environment: 环境参数(温湿度、气压)
  - energy: 能耗数据
  - safety: 安全监测

- **source**: 数据来源
  - sensor: 传感器自动采集
  - manual: 人工录入
  - derived: 计算派生

- **valueType**: 数据类型(number, string, boolean, object, array)

- **domain**: 应用领域(process/quality/geometry/environment等)

- **sampleModel**: 采样模式
  - continuous: 连续采样
  - perStep: 每工步采样
  - perPiece: 每工件采样
  - onEvent: 事件触发

- **allowedRange**: 正常值域范围
- **accuracy**: 测量精度要求

**衍生数据支持**:
- isDerived: 标识是否为计算派生数据
- derivedFrom: 指向源数据的Modality
- derivationRule: 派生计算规则(如min/max/avg/公式)

**数据驱动能力**:
- 统一的数据语义定义
- 数据质量规则库
- 测量点标准化配置
- 跨场景的数据复用

---

### 2.5 ModalityBinding (模态绑定)

**定义**: 将Modality绑定到具体TwinObject,定义该设备上该数据点的具体采集配置

**核心信息**:
- **twinId**: 关联的TwinObject(在哪个设备上)
- **modalityId**: 关联的Modality(采集什么数据)
- **pointRef**: 设备内部的数据点引用(如PLC地址、OPC UA节点)
- **sampleRate**: 实际采样频率
- **unitCode**: 实际使用的单位
- **calibration**: 校准信息(offset, scale, lastCalibrationDate)

**数据驱动能力**:
- 设备级数据采集配置
- 测点-设备映射关系
- 数据采集参数管理
- 校准状态追踪

---

### 2.6 ModalData (模态数据/运行时数据)

**定义**: 一条具体的数据记录或测量值,承载数字孪生系统的运行时数据

**三锚点定位**:
- **refTwin**: 该数据由哪个TwinObject产生(如焊机、工件)
- **refScene**: 该数据在哪个Scene中产生(工序实例)
- **refModality**: 该数据的类型是什么(引用Modality)

**核心信息**:
- **value**: 实际测量值(类型由Modality.valueType决定)
- **observedAt**: 观测时间戳
- **qualityTag**: 数据质量标签(Good/Bad/Uncertain)
- **provenance**: 数据来源(dataSourceId, collectorId, method)
- **sequenceNumber**: 同一Scene内的序列号(用于时序排序)

**数据驱动能力**:
- 全流程数据追溯(通过三锚点)
- 质量数据分析
- 工艺参数监控
- 异常检测与预警
- 时序数据分析

---

### 2.7 组织与人员体系

#### OrgUnit (组织单元)
- 定义组织结构(车间、班组、产线)
- 包含管理资产、产品范围、KPI指标
- 支持组织树形结构(partOf关系)

#### Role (角色)
- 定义岗位角色(班组长、操作工、质检员)
- 包含职责清单、权限集合、任职资质要求
- 关联KPI所有权

#### Assignment (岗位指派)
- 实现人员与角色的时态化绑定
- 记录指派有效期(validFrom, validTo)
- 支持班次轮换、临时顶岗

**数据驱动能力**:
- 人员履职追溯
- 权限控制
- 技能匹配
- 班次管理
- 责任流分析

## 三、数据关联关系图谱

### 3.1 核心关联关系

```
MBOM (工艺定义)
  ├─ refProduct → Product (产品)
  ├─ Route
  │   └─ composedOfTakts → Takt[]
  │       └─ composedOfProcesses → Process[]
  │           └─ composedOfSteps → Step[]
  │               ├─ measurements → [Modality引用]
  │               └─ resourceClasses → [资源类别]
                         ↓ 实例化
Scene (执行实例)
  ├─ refMBOM → MBOM (关联工艺标准)
  ├─ inputWorkpiece → Workpiece (输入工件)
  ├─ outputWorkpiece → Workpiece (输出工件) ← 形成工件流转链
  ├─ involvesAsset → [TwinObject] (参与资产)
  ├─ refFinalProduct → Product (最终产品)
  └─ 产生 → ModalData[] (运行时数据)
                         ↓
ModalData (运行时数据)
  ├─ refTwin → TwinObject (数据生产者)
  ├─ refScene → Scene (业务上下文)
  └─ refModality → Modality (数据类型定义)
                         ↑
Modality (数据字典)
  ├─ 被引用于 ← MBOM.Step.measurements
  └─ 被绑定于 ← ModalityBinding
                         ↓
ModalityBinding (数据点配置)
  ├─ twinId → TwinObject (具体设备)
  └─ modalityId → Modality (数据类型)
```

### 3.2 工件流转链

```
Product (产品定义: M670-SN001)
  ↓ basedOnMBOM
MBOM Route (完整工艺路线)
  ↓ 实例化
Scene Chain (场景链)
  Scene[Takt01] → Scene[Takt02] → ... → Scene[Takt11]
    ↓               ↓                       ↓
  Workpiece_T01 → Workpiece_T02 → ... → Workpiece_T11
  (通过Scene.inputWorkpiece/outputWorkpiece建立流转链)
```

### 3.3 人员责任流

```
OrgUnit (组织单元)
  ├─ contains → OrgUnit[] (子组织)
  └─ defines → Role[] (定义角色)
                ↓
Role (岗位角色)
  ├─ responsibilities (职责清单)
  ├─ permissionSet (权限集合)
  └─ qualificationReq → [Qualification] (资质要求)
                ↓ 指派
Assignment (岗位指派)
  ├─ assigneeId → Person (人员)
  ├─ roleId → Role (角色)
  └─ validFrom/validTo (有效期)
                ↓ 执行
Scene.stepLog (执行记录)
  ├─ performedBy.personId → Person
  ├─ performedBy.roleId → Role
  └─ duration, resultData (执行结果)
```

## 四、数据查询与分析模式

### 4.1 基于三锚点的数据定位

**模式**: `TwinObject × Scene × Modality → ModalData[]`

**示例场景**:
- 查询某台焊机(Twin)在某次焊接工序(Scene)中的焊接电流(Modality)
- 查询某个工件(Twin)在某个检测工序(Scene)中的质量尺寸(Modality)

**SQL伪代码**:
```sql
SELECT * FROM ModalData
WHERE refTwin.object = 'urn:ngsi-ld:TwinObject:Robot:ROBOT-01'
  AND refScene.object = 'urn:ngsi-ld:Scene:Process:M670-SN001:T01:P0010:20251027090000'
  AND refModality.object = 'urn:ngsi-ld:Modality:WeldingCurrent_Auto'
```

### 4.2 基于工件流转链的追溯

**模式**: 通过Scene的inputWorkpiece/outputWorkpiece构建有向图

**示例场景**:
- 追溯某个成品侧墙的完整制造履历
- 分析某批次工件的工序流转路径

**图遍历伪代码**:
```python
def trace_workpiece_chain(product_id):
    scenes = []
    current_scene = find_first_scene(product_id)
    while current_scene:
        scenes.append(current_scene)
        next_workpiece = current_scene.outputWorkpiece
        current_scene = find_scene_with_input(next_workpiece)
    return scenes
```

### 4.3 实际工时 vs 标准工时分析

**模式**: Scene.timeFrame.actualDuration vs MBOM.Step.stdTime

**示例场景**:
- 计算工序效率(OEE)
- 识别瓶颈工位
- 标准工时优化

**分析逻辑**:
```python
def analyze_efficiency(scene):
    mbom_step = get_mbom(scene.refMBOM)
    std_time = mbom_step.stdTime.value
    actual_time = scene.timeFrame.actualDuration
    efficiency = std_time / actual_time * 100
    return efficiency
```

### 4.4 质量数据关联分析

**模式**: ModalData(质量) ← Scene ← MBOM.Step.measurements

**示例场景**:
- 分析焊接参数(工艺)与焊缝质量(结果)的关联
- 识别影响质量的关键工艺参数

**分析路径**:
```
Step.measurements定义质量检测点
  ↓
Scene实例化执行
  ↓
ModalData记录实际测量值
  ↓
质量合格率 = COUNT(qualityTag='Good') / COUNT(*)
```

### 4.5 资源利用率分析

**模式**: TwinObject(设备) ← Scene.involvesAsset

**示例场景**:
- 计算设备稼动率
- 分析人员工作负荷
- 优化资源配置

**计算逻辑**:
```python
def calculate_utilization(twin_id, time_range):
    scenes = find_scenes_with_asset(twin_id, time_range)
    total_work_time = sum([s.timeFrame.actualDuration for s in scenes])
    total_time = time_range.duration
    utilization = total_work_time / total_time * 100
    return utilization
```

## 五、典型业务应用的数据支撑

### 5.1 产线计划可行性验证

**数据支撑**:
- **MBOM**: 提供标准工艺路线、标准工时、资源需求
- **TwinObject(Station/Position)**: 提供产线能力、工位配置
- **Scene历史数据**: 提供实际执行时长、设备可用性统计
- **OrgUnit/Person/Assignment**: 提供人员排班、技能匹配

**验证逻辑**:
1. 基于MBOM标准工时计算理论产能
2. 基于Scene历史数据修正实际工时
3. 基于TwinObject资源检查设备能力
4. 基于Assignment检查人员配置
5. 生成可行性报告与风险预警

### 5.2 产线价值流分析

**数据支撑**:
- **MBOM**: 定义价值流各环节(F1功能)
- **Scene**: 记录实际流转时间(增值时间vs等待时间)
- **Workpiece流转链**: 追溯工件流动路径
- **ModalData**: 提供过程数据支撑

**分析维度**:
1. 增值时间占比(VA Time / Total Time)
2. 工序间等待时间分析
3. 瓶颈工位识别(基于Scene.actualDuration)
4. 物流路径优化(基于Position.transferConnections)

### 5.3 产线自制品管控

**数据支撑**:
- **Workpiece**: 标识每个在制品实体
- **Scene.inputWorkpiece/outputWorkpiece**: 追溯工件位置与状态
- **Scene.status**: 识别在制品状态(InProgress/Waiting)
- **MBOM**: 提供标准生产周期

**管控能力**:
1. 实时在制品盘点(基于Scene.status='InProgress')
2. 在制品账龄分析(当前时间 - Scene.timeFrame.start)
3. 异常滞留预警(超过MBOM标准周期)
4. 工件位置追溯(最后一次Scene.location)

### 5.4 侧墙生产质量提升

**数据支撑**:
- **Modality(quality)**: 定义质量检测指标
- **ModalData**: 记录实际质量测量值
- **MBOM.Step.measurements**: 定义质量控制点
- **Scene**: 提供质量数据的业务上下文

**分析能力**:
1. 质量趋势分析(基于ModalData时序)
2. 工艺参数-质量关联(过程Modality vs 质量Modality)
3. 工位质量对比(不同Station/Position的质量差异)
4. 人员质量影响(Scene.stepLog.performedBy vs 质量结果)
5. 批次质量追溯(基于Workpiece.batchNo)

## 六、数据完整性与一致性保障

### 6.1 强制引用关系

以下引用关系必须保持一致:

1. **Scene ← MBOM**
   - Scene.refMBOM必须指向有效的MBOM实体
   - Scene.sceneLevel必须与MBOM层级匹配

2. **ModalData ← 三锚点**
   - refTwin, refScene, refModality都必须指向有效实体
   - 三者组合必须符合业务逻辑(如焊接电流只能来自焊机)

3. **Workpiece流转链**
   - Scene.outputWorkpiece必须是下一个Scene.inputWorkpiece
   - 不允许出现断链

4. **MBOM ← Modality**
   - Step.measurements中引用的modalityCode必须存在对应Modality

### 6.2 版本管理

- **MBOM版本**: 通过refMBOMVersion标识
- **Scene与MBOM版本**: Scene必须记录执行时使用的MBOM版本
- **历史追溯**: 保留历史版本数据,支持版本对比

### 6.3 时态一致性

- **Assignment有效期**: 确保Scene执行时的人员指派有效
- **Scene时间序列**: 工件流转链中Scene时间必须递增
- **ModalData观测时间**: 必须在Scene时间范围内

## 七、数据规模与性能考虑

### 7.1 数据规模估算(以单条侧墙为例)

- **MBOM**: 1个Root, 1个Route, ~11个Takt, ~20个Process, ~80个Step
- **Scene**: 每条侧墙产生 ~80个Process级Scene, ~200个Step级Scene(如果记录)
- **ModalData**: 每个Scene产生 10-100条数据,单条侧墙约 10,000-50,000条数据
- **Workpiece**: 11个工序实例(每个Takt输出一个Workpiece)

### 7.2 查询优化建议

**高频查询场景**:
1. 基于Scene查询ModalData (refScene索引)
2. 基于TwinObject查询参与的Scene (involvesAsset索引)
3. 基于Product追溯完整工件链 (refFinalProduct + 图遍历)

**建议索引**:
- ModalData: (refScene, refModality, observedAt)
- Scene: (refMBOM, refFinalProduct, status, timeFrame.start)
- TwinObject: (subType, functionCategory)

### 7.3 数据归档策略

- **热数据**: 近3个月的Scene和ModalData (支持实时分析)
- **温数据**: 3-12个月 (支持历史对比)
- **冷数据**: 12个月以上 (归档存储,按需恢复)

2. **Scene.context**: 支持场景级别的扩展信息
3. **ModalData.metadata**: 支持数据级别的自定义标签
4. **Modality.relatedModalities**: 支持数据间关联关系扩展

### 8.2 新业务场景接入

**接入步骤**:
1. 定义新的Modality (如新增检测指标)
2. 在MBOM.Step.measurements中引用
3. 创建ModalityBinding (绑定到设备)
4. Scene执行时产生ModalData
5. 基于新数据开发业务应用

**无需改动**: 核心数据模型保持稳定,新业务通过配置即可接入

## 九、关键技术约束与注意事项

### 9.1 NGSI-LD标准约束

- 所有实体ID必须采用URN格式: `urn:ngsi-ld:{Type}:{Identifier}`
- Property和Relationship类型严格区分
- @context上下文必须包含标准链接

### 9.2 数据质量要求

- **ModalData.qualityTag**: 必须准确标识数据质量(Good/Bad/Uncertain)
- **Scene.status**: 必须及时更新执行状态
- **Workpiece流转链**: 必须保持完整性,不允许断链

### 9.3 安全与权限

- 基于Role.permissionSet实现RBAC访问控制
- 敏感数据(如人员信息)需要脱敏处理
- 数据修改必须记录审计日志(modifiedAt, modifiedBy)
